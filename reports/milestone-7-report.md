## 1. Summary

Milestone 7 delivers the **PDF report generator** using mPDF. A participant can
download their current progress — or completed assessment — as a branded PDF
directly from the dashboard. The report is rendered from the same `ScoringService`
data that powers the dashboard and domain pages: one source of truth for all numbers.

The implementation adds 4 new files, modifies 4 existing files, and extends
`EventRepository` with two new methods for repeatable event tracking.

## 3. Feature coverage (against spec)

| Spec section | Behaviour | Implementation |
|---|---|---|
| **ReportGenerator** | Static `generate($assessment, $mode)` with stream and save | Renders template via `ob_start`/`include`, builds PDF via `STRING_RETURN`, delivers manually (headers + echo for stream, `file_put_contents` for save) |
| **mPDF config** | A4, `dejavusans`, temp dir protected | Margins 15/15/20/20mm, `tempDir` at `uploads/issac-tmp/` with `index.php` + `.htaccess` |
| **Event recording** | `pdf_generated` updates timestamp on every download | `recordOrTouch()` uses `INSERT … ON DUPLICATE KEY UPDATE`; called after successful PDF delivery, not before |
| **Cover** | Title, participant name, date, completion, version, status | Table layout, all text escaped, status formatted from DB value |
| **Summary table** | One row per domain + overall row | Average and band show `—` when `answered === 0` |
| **Progress bars** | Horizontal bar per domain | `<div>` with inline `width` style, clamped 0–100% |
| **Per-domain detail** | Subsection headings, item table (code, prompt, score, band) | Active items in tree order; inactive-but-answered items appended greyed with "(item retired)" |
| **Footer** | "Generated by ISSAC Assessment Platform · {date}" | Via `$mpdf->SetFooter()` with mPDF date token |
| **REST endpoint** | `GET /issac/v1/report`, 400 on empty, 401 on logged-out | Uses shared `permissionCallback()` and `resolveAssessment()` |
| **Dashboard button** | Enabled `<a>` when `answered >= 1`, disabled `<button>` when 0 | Link via `wp_nonce_url(rest_url(...), 'wp_rest')`; label switches at 100% |
| **Last report date** | Displayed below button after first download | `EventRepository::lastFired()` queried in shortcode, formatted via `wp_date()` in site timezone |
| **Save mode** | Writes to `uploads/issac-reports/{user_id}/` | Implemented minimally for Stage 2 readiness; not exercised by user-facing features |

## 4. Deviations from plan (recorded in `ISSAC-development-plan.md`)

1. **PDF delivery uses `STRING_RETURN` instead of `Destination::DOWNLOAD`** — the plan
   says `$mpdf->Output('...', 'D')` for streaming. The implementation renders to a
   string first via `Destination::STRING_RETURN`, then sends headers and body manually.
   This ensures the `pdf_generated` event is recorded only after the PDF is fully
   generated and delivered — if `WriteHTML()` or `Output(STRING_RETURN)` throws, no
   event is recorded and the dashboard never shows a false "Last report" date.

2. **Footer is hardcoded, not ACF-configurable** — plan §8 mentions "configurable footer
   (ACF options page)". The implementation uses a hardcoded string in `SetFooter()`.
   An ACF options page for footer customisation is polish scope (M9), not M7.

3. **No `Plugin.php` changes** — the spec anticipated this: "No new boot call is needed
   if `RoutesController::register()` already handles route registration." Confirmed —
   `ReportEndpoint::register()` is called from `RoutesController::registerRoutes()`,
   which is already wired in `Plugin::boot()`.

4. **Dashboard smoke test updated in-place** — section 6 changed from asserting the
   button is disabled with no PDF route, to asserting the link points to
   `/issac/v1/report` and "Coming soon" text is removed.

## 5. Architecture

```
User clicks "Download report" on dashboard
  → GET /wp-json/issac/v1/report?_wpnonce=…
    → RoutesController::permissionCallback()     (401/403 gate)
    → ReportEndpoint::download()
      → RoutesController::resolveAssessment()    (user_id from session)
      → ResponseRepository::forAssessment()      (400 if empty)
      → ReportGenerator::generate($assessment, 'stream')
        → InstrumentRepository::tree()           (cached)
        → ResponseRepository::forAssessment()
        → ScoringService::summary()              (single source of truth)
        → ob_start() + include templates/report.php
        → Mpdf::WriteHTML() + Output(STRING_RETURN)
        → headers + echo $pdfContent
        → EventRepository::recordOrTouch()       (after delivery)
        → exit
```

## 8. Security

- User resolved from `get_current_user_id()` via `RoutesController::resolveAssessment()`.
- `permissionCallback()` returns 401 (anonymous) or 403 (missing cap).
- All template output escaped: `esc_html()` for text, `esc_attr()` for attributes.
- All SQL via `$wpdb->prepare()` with `absint()` for IDs.
- mPDF temp dir and report save dir protected with `index.php` + `.htaccess Deny from all`.
- Dashboard download URL uses `wp_nonce_url(rest_url(...), 'wp_rest')`.

## 9. Manual verification checklist

**1. Dashboard button active:** log in as a participant with at least 1 response,
visit `/assessment/` — the download button should be an enabled link, not greyed out.

**2. Partial report:** click the download button — a PDF named `ISSAC-Report-{date}.pdf`
should download. Verify:
- Cover shows participant name, date, overall completion %, instrument version.
- Summary table lists all 5 domains with correct answered counts.
- Domains with 0 answers show "—" for average and band.
- Domains with answers show the correct average and band.
- Progress bars reflect completion percentages visually.
- Per-domain detail shows item prompts with scores or "—".

**3. Complete report:** complete all items in one domain, download again — verify the
completed domain shows 100% bar and all items scored.

**4. Fresh user (0 answers):** the download button should be disabled.
Manually hitting `GET /wp-json/issac/v1/report` should return a 400 error.

**5. Button label:** while incomplete: "Download progress report"; at 100% overall:
"Download report".

**6. PDF event + last-generated date:** after first download, reload `/assessment/` — a
"Last report: {date}" line should appear below the download button. Download again,
reload — the date should update. Check `wp_issac_events` — exactly one `pdf_generated`
row with `created_at` reflecting the latest download.

**7. Access gate:** log out and hit the report URL directly — should get 401.

**8. File integrity:** open the downloaded PDF in a viewer — it should not be corrupted.

**9. Mobile:** verify the download button is reachable on a narrow viewport.

**10. Console clean:** no 404s or JS errors on the dashboard after the button is wired.

## 10. Follow-ups / notes for later milestones

- **Admin "download their PDF" button (M8):** `ReportGenerator::generate()` accepts an
  `$assessment` object (not a user ID), so the admin user-detail page can call it
  directly with any user's assessment. The `save` mode is ready for this.
- **Configurable footer (M9):** the footer text is hardcoded in `ReportGenerator`. An
  ACF options page could feed a custom string — trivial to wire when needed.
- **PDF archive (Stage 2):** the `save` mode writes to
  `uploads/issac-reports/{user_id}/`. Adding an `issac_reports` table to track history
  requires only an INSERT after the `file_put_contents` call.
- **Event timing:** `recordOrTouch()` is called after the PDF is fully generated and
  delivered (via `STRING_RETURN` + manual headers). If mPDF throws during rendering, no
  event is recorded.
